Project Overview

A web application enabling legal professionals to upload commercial contracts for AI-assisted analysis. Each document is processed by an external AI service that segments the contract into chunks and returns a structured analysis result (Yes / No / Unknown) per chunk. Lawyers can review, edit, and finalise these results, generate LLM-powered summaries and discussion proposals, and export the completed analysis as a formatted PDF.

The project is delivered in two phases: a core build validated by a small group of internal beta testers, followed by a targeted feedback implementation round.

2
Recommended Technology Stack
Backend / Database / Storage
Convex

Fully managed backend-as-a-service with built-in real-time queries, transactional mutations, and file storage. Postgres-compatible data layer. Chosen over Supabase specifically for its native real-time reactivity, which directly supports the planned multi-user collaborative editing feature without architectural rework later.

Authentication
Convex Auth

Integrated auth layer within Convex. Supports Google OAuth and Microsoft OAuth out of the box, satisfying the SSO sign-in requirements. Eliminates the need for a separate third-party auth service and reduces integration surface area.

Frontend
React + Vite

As specified. Vite provides a fast development experience. React's ecosystem covers all required UI components — PDF rendering (react-pdf), data tables, rich text editing, and PDF generation (react-pdf/renderer).

Hosting / CI
Vercel

Frontend hosting with automatic deployments from Git. Pairs naturally with Vite + React. Preview deployments per branch allow client-facing demos during development without disrupting production.

Architectural note — future multi-user collaboration: Although real-time collaborative editing is out of scope for Phase 1, this is a known future requirement. Convex is selected partly for this reason: its live query model means concurrent editing can be layered on top of the existing data layer without a platform migration. Additionally, all chunk-level updates in Phase 1 will be designed at individual chunk granularity rather than document-level saves, ensuring the data model is ready for per-user, per-chunk concurrency controls when the time comes.

Risks & Mitigation
!
External APIs still under development
All three external APIs (AI analysis, LLM summary, LLM discussion) are still being built and their specs may change. This is the single largest risk to the estimate. Hours for these integrations include a buffer, but significant spec changes mid-development could require additional time. Mitigation: agree on a stable API contract (even a draft schema) before development begins on each integration, and treat breaking spec changes as a scope change event with a written change order.
!
Async AI analysis delivery method TBD
It is not yet clear whether the AI analysis service will deliver results via a synchronous response, a webhook callback, or a polling mechanism. The implementation approach — and its complexity — differs across these options. Webhook handling in particular requires additional infrastructure (a public endpoint, retry logic, idempotency). This will be confirmed with the client's API team before development starts on that module.
i
Data sensitivity and compliance
The application will store commercial contract data, which is legally sensitive. No specific compliance requirements (e.g. data residency, ISO 27001, Australian Privacy Act obligations) have been raised in scope discussions. If the client's firm has compliance obligations — particularly relevant for law firms — these should be scoped separately before development begins, as they may affect infrastructure, logging, and data retention design.
→
Phase 2 scope depends on feedback quality
The Phase 2 estimate assumes informal feedback from ~5 users covering UI/UX refinements and behaviour adjustments within existing features. If beta testing surfaces significant new feature requests or reveals fundamental UX issues requiring redesign, Phase 2 hours may exceed the upper estimate. A scoping session at the start of Phase 2 will establish a firm budget before any work begins.

Assumptions
  • The client will provide access to all three external API endpoints (with sandbox/test credentials) before development of the respective modules begins.
  • UI requirements are minimal — functional and clean, not a bespoke design system. No dedicated UI/UX design engagement is included in this quote.
  • PDF export is a download only. The user is responsible for distributing the file via email. No in-app email sending is in scope.
  • The application is single-tenant (one organisation per deployment). Multi-tenancy, white-labelling, or firm-level administration panels are out of scope.
  • Real-time collaborative editing (multiple users editing simultaneously) is a future-phase feature. Phase 1 will support one user editing at a time per document, with the data model designed to accommodate concurrency later.
  • Content of the LLM prompts (system prompts, context passed to the summary and discussion APIs) is the client's responsibility. Integration assumes a defined request/response format.
  • Testing is manual. Automated test suites (unit, integration, e2e) are not included in this quote but can be scoped separately if required.
  • All infrastructure and SaaS subscription costs listed in Section 6 are paid directly by the client.
